# Live Poker Recorder Architecture

## Purpose

ライブ NLH キャッシュゲーム中にスマホで全ハンドを低い認知負荷で記録し、あとから補完せずに GTO Wizard へ貼り付けられる完全なハンド履歴テキストを生成できる、ローカルファーストな個人用 Web アプリを作る。

中核は分析機能ではなく、実卓の進行を破綻なく記録することにある。ユーザーに勝敗、ポット、サイドポット、合法アクションを計算させず、入力された事実から決まるものはアプリが計算する。

## Context

対象はスマホで使うライブポーカー記録アプリであり、オンラインポーカーや収支分析アプリではない。記録画面は PokerNow のような卓型のゲーム UI を目指し、フォームを縦に並べる画面ではなく、現在必要な入力だけを出す。ハンド記録中の画面はスクロールなしで収まり、重なりや破綻が起きないことが重要である。

初期実装は TypeScript、React、Vite のローカルファースト SPA を前提にする。保存は単一端末・単一ブラウザ内に限定し、アカウント、クラウド同期、複数端末、競合解決は扱わない。ブラウザを閉じても進行中セッションへ戻れることは必要だが、複雑な永続化基盤は作らない。

プリセットは店とレートを階層化せず、名前付きの単一プリセットとして扱う。プリセットには、プリセット名、SB、BB、アンティ有無/額、通貨、チップ単位、標準人数を含める。プレイヤー、座席、自分の席、開始スタックはセッション開始時に設定する。セッション中はプレイヤー名/識別子、着席/離席、スタック補充、キャッシュアウト、席移動、自分の席をハンド間で変更できるが、進行中ハンド中は変更しない。

通常進行の NLH キャッシュゲームを対象にし、プリフロップからリバーまでのフォールド、チェック、コール、ベット、レイズ、オールイン、ブラインド、アンティ、ポット、スタック、ボード、ショーダウンを扱う。複数人オールインとサイドポットは実卓で自然に発生するため対応範囲に含める。ストラドル、ランイットトゥワイス、チョップなど、テーブルルールや例外運用に寄るものは初期範囲外にする。

エクスポートは GTO Wizard 側の入力 UX まで作り込まず、コピペできる PokerStars 風のテキストハンド履歴生成までを初期目標にする。統計分析、クラウド同期、共有、セッション履歴の高度な管理、プリセットのインポート/エクスポートは初期範囲外である。

ポーカーライブラリは `poker-ts` を有力候補として仮定する。ただし、まだライブラリに設計を固定しすぎない。`poker-ts` がアプリの要件に合わない場合は、手札評価に `pokersolver` などの実績あるライブラリを使い、状態機械や精算を自前実装に切り替えられる境界を残す。

## Direction

アーキテクチャは UI から独立した `domain` を中心に置く。`ui`、`storage`、`export`、`poker-library-adapter` は `domain` に依存してよいが、`domain` が React、IndexedDB、ブラウザ API、外部ポーカーライブラリの型や内部状態に依存しないようにする。

`domain` は巨大な reducer にせず、少なくとも `session`、`table`、`hand`、`rules`、`settlement`、`export-model` のように責務を分ける。セッション設定、テーブル状態、ハンド進行、合法アクション、サイドポット精算、エクスポート用表現は変更理由が違うため、小さな純粋関数で検証できる構造にする。

永続化される正は、`schemaVersion` 付きのドメインイベントログとする。現在のスタック、ポット、合法アクション、勝者、サイドポット配分などは保存された状態として扱わず、イベントログから再構築できる派生状態にする。実行中に `AppState` のようなキャッシュを持つのはよいが、正として保存しない。

イベントログは UI 操作ではなく、ポーカー/セッション上の意味を持つドメインイベントで構成する。例として `PlayerSatIn`、`PlayerLeft`、`StackAdjusted`、`HandStarted`、`HoleCardsRecorded`、`ActionTaken`、`BoardCardsDealt`、`ShowdownCardsRevealed`、`HandSettled` のような粒度にする。Undo は直前のユーザー操作に対応するドメインイベントをログから外し、派生状態を再計算する形を基本にする。自動計算された結果を Undo 対象のイベントとして保存しない。

UI はドメインイベントを直接組み立てない。UI は現在状態を表示し、ユーザー意図を `commands` 層へ渡す。`commands` は現在状態と意図から合法性を確認し、保存可能なドメインイベントを生成する。たとえば UI が call の意図を渡し、`commands` が必要コール額や対象 seat を確認して `ActionTaken` を生成する。

外部ポーカーライブラリは `rules`/`settlement` の adapter 層からだけ呼ぶ。`poker-ts` の型や内部状態を保存データ、UI、ドメインイベントへ漏らさない。ライブラリは合法アクション、ポット/サイドポット、ショーダウン計算の候補として使うが、アプリ固有のドメインモデルを主にする。

金額、カード、席、プレイヤーID、ハンドIDは生の `number` や `string` を広く扱い回さず、`ChipAmount`、`CardCode`、`SeatId`、`PlayerId`、`HandId` のような型エイリアスまたは値オブジェクトで境界を作る。通貨やチップ単位の表示は UI/formatting 側に寄せ、domain では比較・加算・減算しやすいチップ額として扱う。

テストは Vitest を前提に、UI よりも domain と commands を厚く検証する。イベントログから状態が再構築できること、合法アクション以外を生成できないこと、ストリート遷移、スタック/ポット、複数人オールイン、サイドポット、ショーダウン、エクスポートを画面なしで確認する。代表的なハンド進行と破綻しやすい all-in/side pot の fixture を持つ。

UI 検証は Playwright で主要画面のモバイル viewport を確認する。大量の E2E ではなく、セッション開始、プリセット選択、1ハンド記録、Undo、エクスポートまでの代表フローに絞る。特に記録画面がスクロールなしで収まること、UI 要素が重ならないこと、画面が破綻していないことを確認する。

## Completion Conditions

- TypeScript + React + Vite のローカルファースト SPA として、スマホで使えるライブ NLH キャッシュゲーム記録アプリが動作する。
- プリセットの作成・編集・削除ができ、名前付きプリセットから SB、BB、アンティ、通貨、チップ単位、標準人数をセッション開始時に選べる。
- セッション開始時にプレイヤー、座席、自分の席、開始スタックを設定でき、ハンド間で着席/離席、スタック補充、キャッシュアウト、席移動、自分の席変更を記録できる。
- ハンド記録画面は卓型のゲーム UI として動作し、モバイル viewport でスクロールなしに収まり、主要 UI 要素の重なりや破綻がない。
- ハンド進行は domain の状態機械と commands を通じて行われ、ルール上可能なアクションだけを記録できる。
- 複数人オールイン、サイドポット、ショーダウン時の勝敗と配分が、入力されたアクション、ホールカード、ボード、ショーダウンカードから計算される。
- 永続化される正は `schemaVersion` 付きドメインイベントログであり、ブラウザを閉じてもセッション、完了済みハンド、進行中ハンド、Undo 可能な直前操作を復元できる。
- Undo は直前のユーザー操作に対応するイベントを取り消し、保存済み派生状態を直接編集せずに現在状態を再構築する。
- 完了済みハンドから、GTO Wizard へコピペするための PokerStars 風テキストハンド履歴を生成できる。
- Vitest で domain と commands の主要ケースが検証され、合法アクション、イベントログ再構築、ストリート遷移、スタック/ポット、サイドポット、ショーダウン、エクスポートの代表ケースが通る。
- Playwright でモバイル主要画面と代表フローが確認され、セッション開始、ハンド記録、Undo、エクスポート、スクロールなし、画面破綻なしが検証される。
